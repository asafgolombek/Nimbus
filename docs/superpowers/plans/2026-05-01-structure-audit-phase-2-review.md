## Technical Explanations

### Why separate PRs for fixes?
Splitting the top-5 fixes into separate PRs (as suggested in Task 10/11) is a best practice for several reasons:
*   **Faster Review:** Smaller, subsystem-scoped PRs are easier for humans to review.
*   **Isolation:** A failure or regression in one fix doesn't block the others from merging.
*   **Clean History:** Each PR has a single, clear purpose ("Fix D10 spawn invariants"), making the git log much more useful.

### Why defer the `_structure.yml` wiring?
Wiring the CI gate too early would cause all current PRs to fail, creating "CI noise." By deferring the wiring until the *last* fix PR (Task 10 Step 3), we ensure that the build only breaks when it's supposed to (i.e., when a *new* violation is introduced).
