### SQLite Schema ‚Äî Unified item (user_version 5)

The authoritative catalog uses a unified **`item`** table (replacing the Q1 **`items`** table) and the **`person`** table. **`LocalIndex.SCHEMA_VERSION` is 5**; `ensureSchema` upgrades older DBs in order (scheduler at user_version 2, unified item at 3, **`person.linked`** at 4, then **Bitbucket / Microsoft / Discord handle columns** at 5).

The SQL below is the **target shape** for Q2 indexing; column names in code may use snake_case in SQLite and camelCase in TypeScript helpers.

**Content indexing scope (Q2):** `body_preview` stores at most 512 characters of stripped text. Full document content extraction (PDF, DOCX, etc.) is explicitly deferred to Q3 (Filesystem connector v2 and embedding pipeline). Q2 FTS5 search therefore covers titles and short previews only.

**Item deduplication:** The `canonical_url` column enables detecting duplicate items when the same content is shared across services (e.g., a file linked in Slack that also lives in OneDrive). The `upsertItem` helper checks for a matching `canonical_url` before inserting and links the duplicate via `metadata.duplicate_of` rather than creating a separate row.

```sql
-- Unified item table (all connectors write here)
CREATE TABLE IF NOT EXISTS item (
  id            TEXT PRIMARY KEY,       -- "{service}:{external_id}"
  service       TEXT NOT NULL,          -- "google_drive", "github", "slack", ‚Ä¶
  type          TEXT NOT NULL,          -- "file", "email", "pr", "message", "issue", ‚Ä¶
  external_id   TEXT NOT NULL,
  title         TEXT NOT NULL,
  body_preview  TEXT,                   -- first 512 chars, stripped of HTML/markdown
  url           TEXT,
  canonical_url TEXT,                   -- normalised URL for cross-service dedup
  modified_at   INTEGER NOT NULL,       -- Unix ms
  author_id     TEXT,                   -- FK ‚Üí person.id
  metadata      TEXT,                   -- JSON blob for service-specific fields
  synced_at     INTEGER NOT NULL,
  pinned        INTEGER NOT NULL DEFAULT 0,  -- 1 = exempt from retention pruning
  UNIQUE(service, external_id)
);

-- FTS5 full-text index over item
CREATE VIRTUAL TABLE IF NOT EXISTS item_fts
  USING fts5(title, body_preview, content="item", content_rowid="rowid");

-- Cross-service people graph
CREATE TABLE IF NOT EXISTS person (
  id                TEXT PRIMARY KEY,   -- deterministic UUID v5 from canonical email or handle key
  display_name      TEXT,
  canonical_email   TEXT UNIQUE,
  github_login      TEXT,
  gitlab_login      TEXT,
  slack_handle      TEXT,
  linear_member_id  TEXT,
  jira_account_id   TEXT,
  notion_user_id    TEXT,
  bitbucket_uuid    TEXT,               -- v5: normalized Bitbucket Cloud user uuid (hex, no braces)
  microsoft_user_id TEXT,               -- v5: Graph user id (Teams channel messages, etc.)
  discord_user_id   TEXT,               -- v5: Discord snowflake user id
  metadata          TEXT,               -- JSON for any additional handles
  linked            INTEGER NOT NULL DEFAULT 1   -- 0 = handle-only, pending manual merge
);

-- Sync state per connector
CREATE TABLE IF NOT EXISTS sync_state (
  service         TEXT PRIMARY KEY,
  cursor          TEXT,                 -- delta token / page token / since timestamp
  interval_ms     INTEGER,              -- persisted per-connector interval override
  last_sync_at    INTEGER,
  status          TEXT,                 -- "ok" | "backoff" | "error"
  error_msg       TEXT,
  next_sync_at    INTEGER
);

-- Per-sync telemetry (append-only; retained for 90 days by the prune job)
CREATE TABLE IF NOT EXISTS sync_telemetry (
  id               INTEGER PRIMARY KEY AUTOINCREMENT,
  service          TEXT NOT NULL,
  started_at       INTEGER NOT NULL,    -- Unix ms
  duration_ms      INTEGER NOT NULL,
  items_upserted   INTEGER NOT NULL DEFAULT 0,
  items_deleted    INTEGER NOT NULL DEFAULT 0,
  bytes_transferred INTEGER,            -- NULL if connector did not report
  had_more         INTEGER NOT NULL DEFAULT 0,  -- 1 if hasMore triggered re-queue
  error_msg        TEXT                 -- NULL on success
);
```

### 1.0 OAuth Client ID Strategy

**Decision:** Nimbus ships with **bundled default OAuth client IDs** for each supported provider (Google, Microsoft, Slack, Notion). These are registered as public/desktop clients with PKCE ‚Äî no client secret is required. Users do not need to create their own app registrations for standard use.

**Rationale:** The project is AGPL-3.0. Bundling client IDs is consistent with other open-source desktop apps (e.g., VS Code, Thunderbird). There is no client secret to protect; the security guarantee is provided by PKCE + the per-user token scoped to the authenticated user's data.

**Custom client ID override:** Power users or self-hosted deployments can override the bundled client ID per provider in `~/.config/nimbus/nimbus.toml`:

```toml
[auth.google]
client_id = "your-own-client-id.apps.googleusercontent.com"

[auth.microsoft]
client_id = "your-azure-app-client-id"
```

**Vault migration:** Credentials stored via DPAPI (Windows), Keychain (macOS), or libsecret (Linux) are tied to the local machine's identity. Moving to a new machine requires re-authentication for all OAuth connectors. The Q4 data export (`nimbus data import`) handles restoring connector registrations but re-triggers the PKCE flow on the new machine to obtain fresh tokens ‚Äî credentials are never transported between machines. Document this clearly in the `nimbus connector auth` help text.

### 1.0.1 Shared provider tokens (Google + Microsoft)

**Live behaviour (Q2):** First-party Google connectors share **one** vault entry; first-party Microsoft connectors share **one** vault entry. There is **no** per-connector OAuth secret in the Vault for these families.

| Provider family | Vault key (JSON access + refresh token) | Connectors using it (Q2) |
|---|---|---|
| Google | **`google.oauth`** | `google_drive`, `gmail`, `google_photos` (and future Google connectors) |
| Microsoft | **`microsoft.oauth`** | `onedrive`, `outlook`, `teams` (and future Microsoft connectors) |

**Overwrite semantics:** Each successful `nimbus connector auth <service>` runs a full PKCE flow and **`set()` replaces** the entire payload for that family‚Äôs key (`google.oauth` or `microsoft.oauth`). The Gateway does **not** merge granted scopes with the previous token. Therefore:

- Authenticating **Drive**, then **Gmail**, with **each connector‚Äôs default scopes only**, causes the **second** flow to **replace** the first token. The new refresh token typically carries **only** the scopes requested in that second auth ‚Äî **Drive may lose `drive.readonly`** until the user re-authenticates Drive or supplies a **union** scope list.
- The same pattern applies to Microsoft: auth **OneDrive** then **Outlook** with defaults can drop **Files** scopes needed by OneDrive.

**User contract (document in CLI help and connector setup guides):**

1. **Prefer a single auth with a union of scopes** when using multiple connectors under the same provider, e.g.  
   `nimbus connector auth google_drive --scopes <drive_scope>,<gmail_scope>,...`  
   using the full HTTPS scope strings (or the connector‚Äôs documented aliases if the CLI accepts them ‚Äî today the IPC path passes scope strings through to Google/Microsoft as configured).
2. **Or** run auth for the **broadest** connector **last** so the final token is a superset (fragile; easy to get wrong).
3. **Or** after adding a connector, **re-run** `nimbus connector auth` for any sibling that stopped working, with `--scopes` listing **all** scopes still required across siblings.

**Implementation reference:** Default scope lists per `serviceId` live in `packages/gateway/src/connectors/connector-catalog.ts` (`oauthProfileForService`). Lazy MCP + REST sync both call `getValidGoogleAccessToken` / the Microsoft equivalent, which read the shared key.

**Future improvement (not Q2):** Optional Gateway behaviour to **merge** new grants into stored tokens, or **separate vault keys per connector** at the cost of more refresh traffic and UX complexity ‚Äî see [Deferred Decisions](#deferred-decisions) (*Per-connector OAuth keys vs shared family key*).

### 1.1 OAuth PKCE Utility (`packages/gateway/src/auth/pkce.ts`)

**Why:** Every Google and Microsoft connector uses OAuth 2.0 with PKCE. Building this once prevents duplicated auth code across 8+ connectors.

**Specification:**

```typescript
export interface PKCEOptions {
  clientId: string;
  scopes: string[];
  redirectPort?: number;          // explicit port; falls back to portRange, then random
  portRange?: [number, number];   // e.g. [49152, 49200]; tries ports in order
  provider: "google" | "microsoft" | "slack" | "notion";
}

export interface PKCEResult {
  accessToken: string;
  refreshToken: string;
  expiresAt: number;              // Unix ms
  scopes: string[];
}

export async function runPKCEFlow(options: PKCEOptions): Promise<PKCEResult>;
export async function refreshAccessToken(
  refreshToken: string,
  provider: "google" | "microsoft",
  clientId: string
): Promise<PKCEResult>;
```

**Implementation notes:**
- Spawn a temporary local HTTP server on `localhost:{redirectPort}` to receive the auth code.
- **Port selection order:** explicit `redirectPort` ‚Üí try each port in `portRange` ‚Üí random ephemeral port (OS-assigned). If a port is in use, try the next. Log a human-readable message to stderr if the random fallback is used so users in restricted environments know to configure `portRange`.
- Document `portRange` in `nimbus.toml` under `[auth]` so corporate/firewall-restricted environments can pin to an approved range without rebuilding.
- Use `crypto.subtle` for PKCE verifier/challenge generation (no external dependency).
- On completion, store tokens via `NimbusVault` using **family** keys: **`google.oauth`** and **`microsoft.oauth`** (full JSON payload per successful PKCE). Older plan examples that used `*.oauth.tokens` are superseded ‚Äî see ¬ß1.0.1.
- Open browser via `PlatformServices.openUrl()` (PAL method ‚Äî platform-specific).
- Tokens are **never** written to stdout, logs, IPC responses, or config files.
- `refreshAccessToken` must write the new token back to the Vault atomically.

**`nimbus connector auth` CLI flags:**

```
nimbus connector auth <service> [--port <n>] [--scopes <scope1,scope2,...>]
```

- `--port` sets an explicit redirect port, overriding config.
- `--scopes` allows the user to request a reduced scope set (e.g., omit `Mail.Read` if only Calendar is needed). The connector validates at runtime that the granted scopes cover its required tool set and disables tools for which scopes are missing ‚Äî it never errors fatally on reduced scopes.

**Tests required:**
- `pkce.test.ts` ‚Äî mock HTTP server; verify PKCE verifier/challenge math; verify Vault write; verify no token leakage in any output path; verify port-range fallback sequence.

### 1.2 Provider Rate Limiter (`packages/gateway/src/sync/rate-limiter.ts`)

**Why:** Google Drive, Gmail, and Google Photos share the same Google API quota. OneDrive and Outlook share the same Microsoft Graph quota. If all three Google connectors fire a sync simultaneously, their individual politeness doesn't prevent collective 429s. The rate limiter is a shared token bucket per provider, acquired before any outbound API call.

```typescript
export type Provider = "google" | "microsoft" | "slack" | "github" | "gitlab"
                     | "bitbucket" | "linear" | "jira" | "notion" | "confluence" | "discord";

export interface ProviderQuota {
  requestsPerMinute: number;
  burstSize: number;           // max tokens that can accumulate
}

// Default quotas ‚Äî conservative defaults; tunable in nimbus.toml under [sync.quotas]
const DEFAULT_QUOTAS: Record<Provider, ProviderQuota> = {
  google:      { requestsPerMinute: 600,  burstSize: 20  },  // 10 req/s across all Google connectors
  microsoft:   { requestsPerMinute: 600,  burstSize: 20  },
  slack:       { requestsPerMinute: 20,   burstSize: 5   },  // Tier 2
  github:      { requestsPerMinute: 83,   burstSize: 10  },  // 5000/hr for PAT
  gitlab:      { requestsPerMinute: 120,  burstSize: 10  },
  bitbucket:   { requestsPerMinute: 60,   burstSize: 5   },
  linear:      { requestsPerMinute: 60,   burstSize: 10  },
  jira:        { requestsPerMinute: 60,   burstSize: 10  },
  notion:      { requestsPerMinute: 30,   burstSize: 5   },
  confluence:  { requestsPerMinute: 60,   burstSize: 10  },
  discord:     { requestsPerMinute: 50,   burstSize: 10  },
};

export class ProviderRateLimiter {
  // Shared singleton ‚Äî instantiated once in the Gateway, injected into SyncContext
  acquire(provider: Provider, tokens?: number): Promise<void>;
  // Awaits until enough tokens are available; tokens defaults to 1.
  // On 429 response: connector calls rateLimiter.penalise(provider, retryAfterMs)
  // which drains the bucket and delays subsequent acquires.
  penalise(provider: Provider, retryAfterMs: number): void;
}
```

**SyncContext update:** Add `rateLimiter: ProviderRateLimiter` to `SyncContext` so every connector can acquire permits without importing a global singleton.

**Connector contract:** All connectors must call `ctx.rateLimiter.acquire(provider)` once per outbound HTTP request batch. Connectors that receive a 429 must call `ctx.rateLimiter.penalise(provider, retryAfterSeconds * 1000)` before retrying.

**`nimbus.toml` override:**
```toml
[sync.quotas.google]
requests_per_minute = 300   # lower if sharing a Google Workspace domain
```

**Tests required:**
- `rate-limiter.test.ts` ‚Äî two mock connectors for the same provider competing for tokens; verify serialization; verify `penalise` delays subsequent acquires by the correct duration.

### 1.3 Delta Sync Scheduler (`packages/gateway/src/sync/scheduler.ts`)

**Why:** All Q2 connectors need periodic background sync. Build this once; connectors implement the `Syncable` interface.

**Interface:**

```typescript
export interface SyncContext {
  vault: NimbusVault;
  db: Database;
  logger: Logger;
}

export interface Syncable {
  readonly serviceId: string;
  readonly defaultIntervalMs: number;
  readonly initialSyncDepthDays: number;  // default: 30; limits first-sync history window
  sync(ctx: SyncContext, cursor: string | null): Promise<SyncResult>;
}

export interface SyncResult {
  cursor: string | null;          // next delta cursor; null if full sync completed
  itemsUpserted: number;
  itemsDeleted: number;
  hasMore: boolean;               // true = more pages pending; scheduler re-queues immediately
  durationMs: number;             // wall-clock time for this sync call
  bytesTransferred?: number;      // optional; connectors that know response sizes should populate this
}

export interface SyncSchedulerConfig {
  maxConcurrentSyncs: number;     // default: 3; prevents CPU/memory spikes on slow machines
  catchUpOnRestart: boolean;      // default: false ‚Äî skip missed syncs, resume normal interval
  retentionDays: number;          // default: 90; items older than this are pruned weekly
}

export class SyncScheduler {
  constructor(config?: Partial<SyncSchedulerConfig>);
  register(connector: Syncable, intervalOverrideMs?: number): void;
  setInterval(serviceId: string, intervalMs: number): void;  // persisted to sync_state
  start(): void;
  pause(serviceId: string): void;
  resume(serviceId: string): void;
  forceSync(serviceId: string): Promise<void>;
  getStatus(serviceId?: string): SyncStatus[];
}
```

**`SyncStatus` shape (consumed by `nimbus connector list/status`):**

```typescript
export interface SyncStatus {
  serviceId: string;
  status: "ok" | "syncing" | "paused" | "backoff" | "error";
  lastSyncAt: number | null;       // Unix ms
  nextSyncAt: number | null;
  intervalMs: number;
  itemCount: number;               // total items indexed for this service
  lastError: string | null;
  consecutiveFailures: number;
}
```

**Concurrency limit:** The scheduler maintains a semaphore of size `maxConcurrentSyncs` (default: 3). When all slots are occupied, pending syncs queue in FIFO order. `forceSync` bypasses the queue and acquires a dedicated slot (max 1 concurrent force-sync per service).

**Initial sync depth:** Every `Syncable` declares `initialSyncDepthDays` (default: 30). On the very first sync (cursor is `null`), the connector only fetches items created/modified within that window. This prevents rate-limit hammering and days-long first syncs for accounts with years of history (e.g., a 10-year Gmail inbox). Users who want full history run `nimbus connector sync --full <service>`, which clears the cursor and performs a depth-unlimited sync in the background.

**Per-connector interval configuration:** Intervals are persisted in `sync_state.interval_ms`. Users set them via `nimbus connector set-interval <service> <duration>` (e.g., `5m`, `1h`, `24h`) or via `nimbus.toml`:

```toml
[sync]
max_concurrent = 3
retention_days = 90         # items older than this are deleted weekly; 0 = keep forever

[sync.intervals]
slack = "5m"
google_drive = "30m"
google_photos = "6h"
notion = "10m"

[sync.initial_depth_days]
gmail = 60                  # override per-connector default of 30
slack = 14
```

**Pinned items:** Items of type `file` and items explicitly starred/saved by the user are exempt from retention pruning regardless of `retention_days`. A `pinned` boolean column is added to the `item` table; connectors set `pinned = true` for items the service marks as starred, important, or saved.

**Offline / catch-up behaviour:** `catchUpOnRestart` defaults to `false`. When the Gateway restarts after downtime, a missed sync is treated as if it had fired at the scheduled time but returned no items ‚Äî the interval clock resets from the current time. This avoids a thundering-herd burst of 13 simultaneous syncs on startup. Users who need aggressive catch-up can set `catchUpOnRestart = true` in config, which runs one (and only one) immediate sync per connector on startup regardless of `next_sync_at`.

**`hasMore` immediate re-queue:** When a sync returns `result.hasMore = true`, the scheduler immediately re-queues the connector without waiting for the next interval tick. The re-queue respects the concurrency semaphore (it does not bypass it) ‚Äî if all slots are occupied, the connector waits in the FIFO queue. This ensures large initial syncs (e.g., 5,000 Gmail messages paged at 100/call) catch up as fast as the concurrency cap and rate limiter allow, not at the interval cadence.

**Backoff policy:**
- Initial retry delay: 5 seconds.
- Multiplier: 2√ó, max: 30 minutes.
- Jitter: ¬±20% of computed delay.
- After 5 consecutive failures: status ‚Üí `"error"`, notify via `PlatformServices.notifications`.

**Telemetry persistence:** Each completed sync (success or error) writes one row to the `sync_telemetry` table (see schema below). This feeds the `nimbus connector status --stats` view.

**Persistence:** All sync state written to `sync_state` SQLite table. Scheduler reads persisted `next_sync_at` on Gateway startup to resume correct intervals.

**Tests required:**
- `scheduler.test.ts` ‚Äî register two mock connectors; verify both fire at correct intervals; verify backoff on failure; verify state persistence across simulated restart; verify concurrency cap (4 connectors, cap=3 ‚Üí only 3 run simultaneously); verify offline/catch-up=false skips missed syncs; verify `hasMore=true` triggers immediate re-queue without waiting for interval.

### 1.5 Schema Migration Runner (`packages/gateway/src/index/migrations/`)

**Status:** **Done** (v1). `LocalIndex.ensureSchema` calls **`runIndexedSchemaMigrations(db, SCHEMA_VERSION)`** in `runner.ts`. Steps still use bundled SQL modules (`schema-sql.ts`, `scheduler-schema-sql.ts`, `unified-item-v3-sql.ts`); **`PRAGMA user_version`** remains the stepping source of truth. A companion table **`_schema_migrations`** records `(version, description, applied_at)` for audit and backfills from `user_version` on existing databases.

**Why:** The jump from v1 (Q1 filesystem-only schema) to v2 (unified `item` + `person` + `sync_state` tables) must be non-destructive. Future quarters will add more migrations. A formal runner prevents hand-rolled `CREATE TABLE IF NOT EXISTS` scattered across the codebase.

**Design (target vs shipped):** The interface below remains the long-term shape; the shipped runner is synchronous and inlined steps rather than per-file `Migration` objects.

```typescript
// Target pattern (future refactor-friendly)

export interface Migration {
  version: number;    // monotonically increasing integer
  description: string;
  up(db: Database): void;   // synchronous, runs inside a transaction
}

export function runMigrations(db: Database): void;
// Called once on Gateway startup via LocalIndex.ensureSchema.
// Steps `user_version` 0‚Üí1‚Üí2‚Üí3; logs each step into `_schema_migrations`.
```

**Shipped layout:**

```
packages/gateway/src/index/migrations/
  runner.ts            # runIndexedSchemaMigrations + ledger/backfill
```

**Invariants:**
- Migrations are append-only. Never modify an already-shipped migration step.
- Each migration step runs in a single SQLite transaction. If it throws, the transaction rolls back and the Gateway logs a fatal error + exits cleanly.
- The **`_schema_migrations`** ledger is created by the runner; do not use it as the primary version driver until/unless we intentionally migrate away from `user_version`.
- Downgrade (`down()`) is not implemented in Q2 ‚Äî schema rollback is handled by restoring a database backup.

### 1.6 Lazy Connector Startup (`packages/gateway/src/connectors/lazy-mesh.ts`)

**Status:** **Done** for the **Google MCP bundle** (Drive + Gmail + Google Photos in one `MCPClient` with three servers). Filesystem MCP (`@modelcontextprotocol/server-filesystem`) stays **always connected** (required for local-first indexing). When Vault contains **`google.oauth`**, `listTools()` / **`ensureGoogleDriveRunning()`** start **all three** subprocesses (shared `GOOGLE_OAUTH_ACCESS_TOKEN` env). **Microsoft bundle** (OneDrive + Outlook + Teams) starts when **`microsoft.oauth`** exists (`MICROSOFT_OAUTH_ACCESS_TOKEN`). **GitHub, GitLab, Bitbucket, Slack, Linear, Jira, Notion,** and **Confluence** MCPs start when their respective vault keys exist (see `lazy-mesh.ts`). **Discord** MCP starts only when **`discord.enabled === "1"`** and **`discord.bot_token`** are set. Used by REST syncables before outbound API work. Idle disconnect defaults to **300_000 ms** per bundle; tool-list epoch bumps so dispatcher caches refresh.

**Why:** Spawning many MCP server child processes at Gateway startup would waste memory for connectors the user doesn't use in a given session, and would increase cold-start time past the 100ms Q1 target.

**Target design (full registry):** Connectors are registered with spawn config but are **not started** until first use. "First use" is defined as:
- A tool call routed to that connector, OR
- A scheduled sync tick firing for that connector.

After `inactivityTimeoutMs` (default: 5 minutes) with no tool calls, the connector process is shut down and its slot is freed. The next tool call or sync tick restarts it.

```typescript
export interface ConnectorEntry {
  serviceId: string;
  spawnConfig: MCPServerConfig;
  inactivityTimeoutMs: number;    // default: 300_000 (5 min)
  state: "stopped" | "starting" | "running" | "error";
  process: BunSubprocess | null;
}
```

**Sync interaction:** Google **`Syncable`** implementations call **`ensureGoogleDriveRunning()`** before REST work so the bundled Google MCP processes can start if needed. As more lazy connectors appear, the pattern should converge on an explicit `ensureRunning(serviceId)` on the mesh used by both the engine and the scheduler.

## Phase 6 ‚Äî People Graph & Cross-Service Identity (Weeks 9‚Äì11)

### 6.1 People Graph (`packages/gateway/src/people/`)

The people graph resolves cross-service identities ‚Äî e.g., "who is the most active reviewer on the payment-service repo and what are they working on in Linear?" ‚Äî without a network call.

**Linker algorithm:**

```
Priority 1: Canonical email match (exact)
Priority 2: GitHub login ‚Üí GitHub API profile email ‚Üí canonical email
Priority 3: Slack handle ‚Üí Slack profile email ‚Üí canonical email
Priority 4: Linear member ‚Üí Linear profile email ‚Üí canonical email
Priority 5: Jira account ‚Üí Jira profile email ‚Üí canonical email
```

**Split-email handling:** A person who uses different emails across services (personal GitHub, work Slack) will produce two separate `person` rows. The linker **never merges solely on name match** ‚Äî email evidence is required. When no email match is found:
- A new person record is created with the service-specific handle.
- The record is marked `linked = false` to signal it is a candidate for manual review.
- `nimbus people list --unlinked` shows all unresolved person records.
- The user resolves them with `nimbus people link <id-a> <id-b>`, which calls `mergePeople` and promotes the surviving record to `linked = true`.

**Implementation:**

```typescript
// packages/gateway/src/people/linker.ts

export async function resolvePersonId(
  handle: string,
  service: "email" | "github" | "gitlab" | "slack" | "linear" | "jira" | "notion"
): Promise<string> {
  // 1. Look up existing person by service handle
  // 2. If found, return person.id
  // 3. Attempt email-based merge with existing records (email evidence only)
  // 4. If no match, create new unlinked person record
  // Returns deterministic UUIDv5 based on canonical_email (or handle if no email)
}

export async function mergePeople(
  db: Database,
  personIdA: string,
  personIdB: string
): Promise<string>;  // Returns the surviving person.id; updates all item.author_id FKs
```

**Privacy controls:**
- The people graph is **entirely local**. It is never synced to a cloud service or transmitted over IPC outside the local machine.
- `nimbus connector remove <service>` removes all `person` records whose **only** indexed items came from that service. Person records that still have items from other services are retained but their service-specific handle (e.g., `slack_handle`) is cleared.
- There are no people-graph entries for individuals the user has never directly interacted with (author, reviewer, assignee, sender, recipient). Passive participants (e.g., users in a Slack workspace but not in any synced message) are not indexed.

**Population strategy:**
- Linker runs during each connector's delta sync ‚Äî after items are upserted, `resolvePersonId` is called for each `author_id` in the batch.
- Full-graph population runs once after all connectors have completed their first sync.
- `mergePeople` is called when a sync discovers two `person` rows with the same `canonical_email`.

**New CLI commands:**

| Command | Description |
|---|---|
| `nimbus people list` | List all known people (cross-service) |
| `nimbus people list --unlinked` | Show person records with no confirmed email match |
| `nimbus people get <id>` | Show all service handles + item count for a person |
| `nimbus people link <id-a> <id-b>` | Manually merge two person records |

**API exposed via Gateway IPC:**

```typescript

"people.search"   // by name, email, or service handle
"people.items"    // all items authored by a person (cross-service)
"people.merge"    // manual merge (called by `nimbus people link`)

```

## Phase 7 ‚Äî CLI, Testing, and Hardening (Weeks 10‚Äì13)

### 7.0 Engine-Layer Additions

Two engine-layer additions are required to make Q2's expanded index useful to the LLM.

#### Context Ranker

**Problem:** A broad query like "find everything I've touched this sprint" may match hundreds of index items. Passing all of them to the LLM hits context window limits and increases latency.

**Solution:** A `ContextRanker` step runs between `searchItems` and the LLM call. It:

1. Takes the full ranked result set from `searchItems`.
2. Selects the top-N items (default: `20`, configurable via `engine.context_window_items` in `nimbus.toml`) as full context objects.
3. Collapses the remainder into a **source summary**: a compact list of `{ service, type, count }` groups (e.g., "32 more GitHub PRs, 14 more Slack messages").
4. Appends the source summary to the LLM prompt with the instruction: "Use `index.fetchMore` if you need details on a specific source."
5. Exposes `index.fetchMore(service, type, offset, limit)` as an agent tool so the LLM can page in more results if needed.

```typescript
// packages/gateway/src/engine/context-ranker.ts
export interface ContextWindow {
  items: Item[];            // top-N full items
  sourceSummary: SourceGroup[];
  totalMatches: number;
}

export interface SourceGroup {
  service: string;
  type: string;
  count: number;
  oldestModifiedAt: number;
  newestModifiedAt: number;
}

export function buildContextWindow(
  results: Item[],
  maxItems?: number          // defaults to config value
): ContextWindow;
```

#### People Resolver Tool

**Problem:** When a user asks "show me PRs from Sarah", the LLM needs to map the name to a canonical `person.id` before querying the `item` table. Without this, the LLM either hallucinates a GitHub login or produces an empty result.

**Solution:** Add `people.resolve` to the Gateway's core agent toolset (registered alongside `searchLocalIndex` in `agent.ts`):

```typescript
// Registered as a Mastra tool in packages/gateway/src/engine/agent.ts
tools: {
  searchLocalIndex:  createSearchLocalIndexTool(),

  getAuditLog:       createAuditLogTool(),

```

`resolvePerson` calls `people.search` (IPC) with the natural-language name string and returns up to 3 candidate `person` objects (id, display_name, known handles). The LLM picks the best match or asks the user to clarify if there are multiple candidates.

### 7.2 Search Ranking Strategy

The `searchItems` helper (¬ß1.3) must return results ranked by a deterministic, tunable formula. For Q2 (FTS5 + structured metadata only ‚Äî no vectors until Q3):

**Ranking formula:**

```
score = (0.5 √ó fts5_bm25_score)
      + (0.3 √ó recency_score)
      + (0.2 √ó service_priority_score)
```

- **`fts5_bm25_score`**: normalised BM25 rank from the FTS5 `rank` function (higher = more relevant).
- **`recency_score`**: `1 / (1 + days_since_modified)` ‚Äî items modified today score ‚âà1.0; items modified a year ago score ‚âà0.003.
- **`service_priority_score`**: configurable per-service weight (default 0.5 for all). Users can raise priority for their most-used services in `nimbus.toml`:

```toml
[search.priority]
github = 0.8
slack  = 0.7
notion = 0.6
```

**Tie-breaking:** Items with equal scores are ordered by `modified_at DESC`.

**Deduplication in results:** Before returning, `searchItems` groups results by `canonical_url`. When two items share a `canonical_url`, only the higher-scoring one is returned, with a `duplicates` field listing the suppressed service IDs. This surfaces "same file linked from Slack and stored in OneDrive" as one result.

### 7.9 Installers (Windows, macOS, Linux)

**Goal:** Ship reproducible **installers** that place the compiled `nimbus` CLI and `nimbus-gateway` binaries in a single install directory (same layout the CLI uses to spawn the gateway: sibling `nimbus-gateway` / `nimbus-gateway.exe` next to the CLI). Tauri desktop bundling (Q4) remains separate; this phase is headless Gateway + CLI only.

**Dependencies:**

- Stable gateway launch resolution in the CLI (`nimbus start` finds sibling binary, monorepo `dist/` build, or `bun run` from source; optional `NIMBUS_GATEWAY_EXECUTABLE` override).
- Agreed install directory convention documented for packagers (e.g. `Program Files\Nimbus\` on Windows, `/usr/lib/nimbus/bin` + `/usr/local/bin` wrappers on Linux ‚Äî see **`installers/README.md`**).

**Deliverables:**

- **Headless bundle (in tree):** `bun run package:headless` copies compiled **`nimbus-gateway`** + **`nimbus`** into **`dist/headless-bundle/`** (after `bun build` from `packages/gateway` / `packages/cli` into repo `dist/`, matching `release.yml`). PR CI runs a Linux smoke build of both binaries then invokes the script.
- **Windows:** **NSIS** script **`installers/windows/nimbus-headless.nsi`** (rationale: single file, no MSI toolchain dependency; install to `%ProgramFiles%\Nimbus\`). Run `makensis` locally or add a Windows workflow job; PATH addition is manual unless extended ‚Äî see **`installers/README.md`**.
- **macOS:** **`installers/macos/build-pkg.sh`** produces an unsigned **`.pkg`** for smoke; **codesigning** and **notarization** use Apple Developer ID secrets documented in **`installers/README.md`** and **`release.yml`** placeholders.
- **Linux:** **`bun run package:installers:linux`** produces **`.deb`** (`/usr/lib/nimbus/bin` + `/usr/local/bin` wrappers) and **`.tar.gz`** (`bin/` layout). PR CI + Ubuntu push matrix + **`release.yml`** attach artifacts under **`dist/installers/`**.
- **CI:** Linux installer build is wired (PR + matrix Ubuntu + release publish). Optional: add `nimbus help` smoke on extracted tarball in CI; Windows/macOS installer **build** jobs remain optional extensions.
- **Secrets / signing:** **`installers/README.md`** ‚Äî certificates live in GitHub Actions secrets only; unsigned artifacts acceptable until release tags.

**Out of scope for this subsection:** Bundling MCP connector server binaries inside the installer (follow-up unless Q2 scope explicitly expands); Tauri app packaging.

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|

## Deferred Decisions

Items raised during review that are acknowledged but deliberately not addressed in Q2.

| Topic | Decision | Reason |
|---|---|---|
| **SQLite DB encryption at rest** | Deferred to post-Q2 | Adding SQLCipher introduces a native dependency that would break the Bun pure-JS build and require platform-specific binaries. OS-level filesystem encryption (BitLocker, FileVault, LUKS) covers the same threat model at zero cost. Revisit if a security audit identifies a gap. |
| **Full document content extraction** (PDF, DOCX body text in FTS5) | Deferred to Q3 | Requires the embedding pipeline and `@xenova/transformers` introduced in Q3. Q2 FTS5 search covers title + 512-char preview. |
| **Vault credential portability between machines** | Deferred to Q4 | Handled by `nimbus data export/import` (Q4 delivery). Q2 connectors must re-authenticate on a new machine. Document this in `nimbus connector auth` help text. |
| **Generic user-defined MCP connector** (`nimbus connector add --mcp "<cmd>"`) | Deferred to Q3 | The Q3 Extension Registry v1 is the correct home for this. It adds manifest hash verification, sandboxed child processes, and scope-limited credential injection ‚Äî all required for safe third-party MCP servers. Shipping a simpler version in Q2 would create a parallel, less-safe code path that must then be migrated in Q3. |
| **Per-connector OAuth vault keys vs shared family key** | **Shared key for Q2** (`google.oauth`, `microsoft.oauth`) | Keeps refresh logic and lazy MCP env injection simple. Alternative: separate keys per `serviceId` (more refresh calls, clearer isolation) **or** Gateway-side **scope merge** on auth (read existing token, request incremental consent, union `scopes` in stored JSON). Revisit if users repeatedly break sibling connectors by re-auth. |
