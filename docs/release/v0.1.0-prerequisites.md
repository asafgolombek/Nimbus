# v0.1.0 Release Prerequisites — Runbook

**Purpose:** Everything that must be procured, generated, or configured outside of the codebase before Nimbus can ship `v0.1.0`. This is a runbook, not a design doc. Work through it in the order below — ordered by real-world lead time, longest first, so the slow things start immediately and fast things fill the gaps.

**Applies to:** Phase 4 release gate. Assumes single-maintainer ownership (`asafgolombek`). For a company/org purchase path, substitute the legal entity name where "individual" is called out.

**Legend:**
- 💰 = out-of-pocket cost (annual unless noted)
- ⏱ = typical lead time
- 🔐 = produces a secret that must be stored in a password manager + GitHub Actions secret
- 🌐 = produces a public artifact (URL, key) that must be committed / announced

## 1. Windows Authenticode Code-Signing Cert (longest lead time)

**Why first:** EV cert procurement includes identity verification (notarized docs or video call), issuance on a hardware token (physical shipment), and the SmartScreen reputation clock starts from first signed binary — the earlier you have it, the sooner SmartScreen warnings clear.

**Decision: OV vs EV.**

| | OV (Organization/Individual Validation) | EV (Extended Validation) |
|---|---|---|
| SmartScreen | Warns until reputation built (~1000s of downloads over weeks/months) | Clears immediately for known-reputation publisher on first signed release |
| Key storage | Software (PFX on disk, or recommended: YubiKey) | Hardware token (mandatory) |
| 💰 Cost | ~$200–400/yr (individual OV harder to get; most CAs only issue org) | ~$350–700/yr for individual EV (some CAs; e.g. SSL.com offers individual EV) |
| ⏱ Lead time | 1–5 business days after docs submitted | 5–15 business days (shipping + verification) |
| Documentation required | Gov ID + address proof | Gov ID + notarized letter or video identity verification |

**Recommendation:** **Individual EV from SSL.com or Sectigo.** SmartScreen warnings on an OV-signed new app kill adoption — you'd ship v0.1.0 and every user would see "Windows protected your PC." EV costs more but is the right call for a public launch.

**Steps:**

1. **Choose a CA.** Options that issue EV to individuals:
   - SSL.com — https://www.ssl.com/certificates/ev-code-signing/
   - Sectigo (formerly Comodo) — https://sectigostore.com/
   - DigiCert — https://www.digicert.com/signing/code-signing-certificates (most expensive, fastest)
   - GlobalSign — https://www.globalsign.com/en/code-signing-certificate
2. **Order under your legal name** (or DBA if you have one registered). The "publisher" displayed in Windows UAC dialogs will be this exact string.
3. **Identity verification:**
   - Prepare: government photo ID (passport preferred), utility bill or bank statement with name + address (<90 days old).
   - Some CAs require a notary visit or video verification. Budget a half-day.
4. **Hardware token ships** (YubiKey 5 FIPS, or CA-branded token). They cannot email you the private key.
5. **Install the CA's client software** (e.g. SSL.com `eSigner`, SafeNet Authentication Client for Thales tokens). You'll need this plus `signtool.exe` from the Windows SDK to sign.
6. **Test sign a dummy exe** locally before wiring CI:
   ```powershell
   signtool sign /tr http://timestamp.sectigo.com /td sha256 /fd sha256 /a dummy.exe
   ```
7. **CI integration (for `release.yml`):** EV tokens don't live in GitHub Actions secrets — they're physical. You have two options:
   - **Self-hosted Windows runner** with the token plugged in (most common for small teams). GitHub Actions can target it via `runs-on: [self-hosted, windows, nimbus-signing]`.
   - **Cloud signing service** (e.g. SSL.com eSigner, Azure Key Vault Certificate with managed signing). Token-free, more expensive per signing call, but fits native GHA.
   - **Recommended for solo maintainer:** SSL.com eSigner (~$20/mo on top of the cert). Purely cloud-based, no physical token handling. Cost is worth it vs. running a self-hosted runner.

**Output to record in the password manager:**
- CA order reference + login
- eSigner username + password + TOTP seed (if using eSigner)
- Cert thumbprint (goes into `release.yml` as a non-secret variable)

**Cross-reference in plan:** Windows `signtool` invocation is expected in `.github/workflows/release.yml`. Runbook delivery is "cert in hand + eSigner credentials in GitHub Actions encrypted secrets."

## 2. Apple Developer ID + Notarization

**Why second:** Enrollment can take 2–5 business days, sometimes longer if Apple flags the application for manual review.

**Decision: Individual vs Organization.**

| | Individual | Organization |
|---|---|---|
| Publisher name shown | Your legal name | Company name |
| 💰 Cost | $99/yr | $99/yr |
| ⏱ Lead time | 1–3 business days | 5–10 business days (D-U-N-S number required — free via Dun & Bradstreet, 5 business days) |
| Legal exposure | Personal | Entity |

**Recommendation:** Individual for v0.1.0. You can migrate to an Organization account later by transferring apps — not free, but not destructive.

**Steps:**

1. **Enroll** at https://developer.apple.com/programs/enroll/.
2. **Install Xcode Command Line Tools** on a Mac (mandatory — notarization tooling is macOS-only). `xcode-select --install`.
3. **In Xcode → Settings → Accounts**, add your Apple ID. Apple will issue two certs you need:
   - **Developer ID Application** (for signing `.app` bundles and standalone binaries)
   - **Developer ID Installer** (for signing `.pkg` installers — required since Phase 4 ships `.pkg`)
4. **Export both certs** to `.p12` files (including private keys). Protect with a strong passphrase.
5. **Create a notarization app-specific password:**
   - https://appleid.apple.com → Sign-In and Security → App-Specific Passwords → Generate.
   - Label it `nimbus-notarytool`.
6. **Record your Team ID** (10-char alphanumeric; top-right of developer.apple.com → Account).
7. **Test sign + notarize locally** before wiring CI:
   ```bash
   codesign --sign "Developer ID Application: Your Name (TEAMID)" \
            --options runtime --timestamp nimbus-gateway
   xcrun notarytool submit nimbus-gateway.zip \
         --apple-id you@example.com \
         --team-id TEAMID \
         --password APP_SPECIFIC_PASSWORD \
         --wait
   xcrun stapler staple nimbus-gateway.pkg
   ```
8. **CI integration (for `release.yml`):**
   - Store in GitHub Actions encrypted secrets: `APPLE_ID`, `APPLE_APP_SPECIFIC_PASSWORD`, `APPLE_TEAM_ID`, `APPLE_DEVELOPER_ID_APPLICATION_P12_BASE64`, `APPLE_DEVELOPER_ID_INSTALLER_P12_BASE64`, `APPLE_P12_PASSWORD`.
   - Base64-encode `.p12` files: `base64 -i DeveloperID.p12 | pbcopy`.
   - GHA `macos-14` runner has Xcode + notarytool preinstalled.

**Output to record:**
- Apple ID used for enrollment
- App-specific password (once only — can't be viewed again)
- Team ID
- Both `.p12` files + passphrase (kept in password manager, base64'd into GHA secrets)

**Cross-reference in plan:** macOS `codesign + notarytool + stapler` invocations in `.github/workflows/release.yml`.

## 3. GPG Signing Key (Linux)

**Why before Ed25519:** Same cryptographic hygiene applies. Generate the master key offline, keep it air-gapped, use subkeys for day-to-day signing.

💰 Free. ⏱ ~30 min.

**Steps:**

1. **On an offline (or freshly booted) machine,** install a recent GnuPG (`gpg --version` ≥ 2.2).
2. **Generate a master key** (Certify-only, no expiry):
   ```bash
   gpg --expert --full-generate-key
   # Select: (11) ECC (set your own capabilities)
   # Toggle off Sign/Encrypt/Authenticate — leave only Certify
   # Curve: Curve 25519
   # Expiry: 0 (never) — master key
   # Real name: Nimbus Release Signing
   # Email: releases@<your-domain>
   ```
3. **Add subkeys:**
   ```bash
   gpg --edit-key releases@<your-domain>
   > addkey  # → ECC Sign only, Curve 25519, expire 2y
   > save
   ```
4. **Export for CI:**
   ```bash
   gpg --armor --export releases@<your-domain> > public.asc
   gpg --armor --export-secret-subkeys releases@<your-domain> > signing-subkey.asc
   ```
   The subkey export excludes the master — even if CI is compromised, the master is safe.
5. **Back up the master key offline.** Copy the entire `~/.gnupg` directory to two USB drives stored separately. **Wipe master key from CI machine.**
6. **Publish public key:**
   - `gpg --keyserver keys.openpgp.org --send-keys <KEYID>`
   - Upload `public.asc` to the repo at `docs/release/SIGNING-KEY.asc` and link it in `README.md`.
7. **CI integration:**
   - GitHub Actions secrets: `GPG_SIGNING_SUBKEY` (contents of `signing-subkey.asc`), `GPG_PASSPHRASE`.
   - In `release.yml`:
     ```yaml
     - run: echo "$GPG_SIGNING_SUBKEY" | gpg --batch --import
     - run: gpg --batch --yes --passphrase "$GPG_PASSPHRASE" --pinentry-mode loopback --detach-sign nimbus_0.1.0_amd64.deb
     ```

**Output:**
- `public.asc` (committed to repo)
- `signing-subkey.asc` + passphrase (GHA secrets)
- Master key in offline backup

## 4. Ed25519 Updater Signing Keypair

**Why:** Auto-update verification. Binary is signed with this key; Gateway verifies with the embedded public key before installing an update. Without this, `NIMBUS_DEV_UPDATER_PUBLIC_KEY` is still in place — which means any dev key can install updates. That is **unshippable**.

💰 Free. ⏱ ~15 min.

**Steps:**

1. **Generate keypair** on the same offline machine as the GPG master:
   ```bash
   openssl genpkey -algorithm ED25519 -out updater-private.pem
   openssl pkey -in updater-private.pem -pubout -out updater-public.pem
   ```
2. **Extract raw 32-byte public key** (what gets embedded in the binary):
   ```bash
   openssl pkey -in updater-public.pem -pubin -outform DER | \
     tail -c 32 | base64
   ```
3. **Replace the embedded public key:**
   - File: `packages/gateway/src/updater/public-key.ts`
   - Swap the current embedded value with the base64 from step 2.
   - Keep the `NIMBUS_DEV_UPDATER_PUBLIC_KEY` override working for tests — don't remove it.
4. **CI integration:**
   - GHA secret: `UPDATER_ED25519_PRIVATE_KEY` (contents of `updater-private.pem`).
   - In `release.yml`, sign each compiled binary with a short script that shells out to `openssl pkeyutl -sign`.
5. **Back up private key offline** alongside GPG master.
6. **Rotation plan:** Document in `docs/release/SIGNING-KEY-ROTATION.md` that key rotation requires shipping one update signed by the *old* key that embeds the *new* public key, before the old key can be retired.

**Output:**
- `updater-public.pem` / raw base64 (committed in source)
- `updater-private.pem` in GHA secret + offline backup

## 5. `registry.nimbus.dev` Registry Host

**Purpose:** Serves `index.json` — the Marketplace catalog of community extensions. No auth, no backend, no logs. Just a signed JSON file.

**Options:**

| | Cloudflare Pages (recommended) | GitHub Pages | Minimal VPS |
|---|---|---|---|
| 💰 Cost | Free tier fits | Free | ~$5/mo |
| TLS | Auto via CF | Auto via GitHub | You configure |
| Custom domain | Native | Native | You configure |
| Signature (anti-tamper) | File signed by GPG subkey | Same | Same |
| Rebuild | Pushed from GH repo | Pushed from GH repo | `scp` |

**Recommendation:** Cloudflare Pages, sourced from a new repo `nimbus-dev/nimbus-registry`.

**Steps:**

1. **Create a new public GitHub repo** `nimbus-dev/nimbus-registry`.
2. **Structure:**
   ```
   nimbus-registry/
     index.json           # catalog
     index.json.asc       # GPG detached signature
     extensions/
       com.example.notion/
         1.0.0.json       # per-version manifest snapshot
     README.md
   ```
3. **Seed `index.json`** with an empty extensions array for v0.1.0:
   ```json
   {
     "schemaVersion": 1,
     "updatedAt": "2026-04-21T00:00:00Z",
     "extensions": []
   }
   ```
4. **CI in the registry repo** — a small GHA workflow that, on push to `main`, re-signs `index.json` with the GPG subkey from section 3 and commits the `.asc`. Same secrets as section 3.
5. **Deploy to Cloudflare Pages:**
   - Cloudflare dashboard → Pages → Create → Connect to Git → pick `nimbus-registry`.
   - Build command: (none — static). Output directory: `/`.
   - Custom domain: `registry.nimbus.dev` → Cloudflare auto-creates DNS + TLS.
6. **Gateway change:** Verify the client code that fetches the registry validates the signature against the GPG public key before parsing. If it doesn't, that's a plan item for the Marketplace workstream.

**Output:**
- `registry.nimbus.dev` live, serving `index.json` over TLS
- Signing automation in the registry repo's CI

**Fallback if Cloudflare Pages is off-limits:** Static files in the main repo under `docs/` + GitHub Pages on the `docs` site at `nimbus.dev/registry/index.json`. Same signature flow.

## 6. VS Code Marketplace Publisher

💰 Free. ⏱ ~30 min.

**Steps:**

1. **Create an Azure DevOps organization** (free) at https://dev.azure.com. Name it `nimbus-dev` if available (this is only the org name, not the publisher).
2. **Create a Personal Access Token:**
   - User Settings → Personal Access Tokens → New Token.
   - Organization: **All accessible organizations** (required — Marketplace is not tied to a single AD org).
   - Scopes: **Custom defined → Marketplace → Manage**.
   - Expiry: 1 year (set a calendar reminder to rotate).
3. **Create the Marketplace publisher** at https://marketplace.visualstudio.com/manage.
   - Publisher ID: `nimbus-dev` (lowercase, used in the extension's `package.json` as `"publisher": "nimbus-dev"`).
   - Display name: `Nimbus`.
   - Accept publisher agreement.
4. **Install `vsce`** on your dev machine: `npm i -g @vscode/vsce`.
5. **Smoke test with the PAT:** `vsce login nimbus-dev` → paste PAT. No publish yet — just verify the token works.
6. **CI integration:** GHA secret `VSCE_PAT` used by `.github/workflows/publish-vscode.yml` which will trigger on `vscode-v*` tags.

**Output:**
- Publisher ID `nimbus-dev`
- Azure DevOps PAT (GHA secret + password manager, with expiry reminder)

## 9. GitHub Repo Configuration

💰 Free. ⏱ ~30 min.

**Steps:**

### 9.1 Branch protection on `main`

Settings → Branches → Add rule for `main`:
- ✅ Require pull request before merging
- ✅ Require approvals: 1 (set to 0 only if you're the only maintainer — revisit when team grows)
- ✅ Dismiss stale approvals on new commits
- ✅ Require status checks to pass: `pr-quality` (and any others from `.github/workflows/ci.yml`, `security.yml`, `codeql.yml`)
- ✅ Require branches to be up to date before merging
- ✅ Require conversation resolution
- ✅ Require linear history (optional but recommended)
- ✅ Do not allow bypassing (including admins — you can uncheck this for solo dev if needed, but default to on)

### 9.2 Dependabot

`.github/dependabot.yml` already exists per architecture.md. Verify:
- npm (root) — daily or weekly
- github-actions — weekly
- Rust (Tauri) — weekly

### 9.3 Actions permissions

Settings → Actions → General:
- Workflow permissions: **Read and write** (needed for release.yml to create releases).
- Allow GitHub Actions to create and approve PRs: Off unless using a bot.

### 9.4 Release environment gating

For high-privilege release workflows:
- Settings → Environments → New environment `release`.
- Required reviewers: yourself.
- Deployment branches: `main` only.
- Wildcard protected tags: `v*`, `client-v*`, `vscode-v*`.

In `release.yml` and `publish-vscode.yml`, set `environment: release` on the publish jobs. You'll approve each release manually from the Actions UI before signing + upload runs. This prevents a stolen GHA token from pushing a release without your button-press.
